Was fehlt
wirklich?

Vier echte Lücken — bewertet nach Grösse des Problems, Schwierigkeit zu bauen, und Risiko dass ein anderer es zuerst macht.

Lücke 1 — Stärkstes Signal

CI/CD Design System Drift-Detector

BAUEN OPEN SOURCE 6–8 WOCHEN

Problem: Wenn Cursor, Copilot oder Claude Code auf einem PR arbeitet, weiss heute niemand ob der generierte Code die Design-System-Tokens und Komponenten korrekt nutzt — oder ob er ad-hoc Farben und Abstände hardcoded. Das passiert täglich, in jedem Team das AI-Coding nutzt.

Was fehlt: Ein GitHub Action / GitLab CI Plugin, das auf jedem PR scannt und flaggt: "Diese Zeile nutzt #3B82F6 statt color.primary.500". Ähnlich wie Salesforce's Lightning Linter, aber framework-agnostisch und open-source.

Markt-Lücke

Kein generisches Open-Source-Tool existiert heute. Salesforce hat eines für ihr eigenes System — niemand hat das Problem allgemein gelöst.

Bauaufwand

Moderat. CSS/JS-Linting ist gelöstes Problem. Design-Token-Integration + Figma API = 6–8 Wochen für ein funktionsfähiges MVP.

Strategie

Open Source auf GitHub = Community, Reputation, Lead Gen. WeSlam monetarisiert mit Setup-Service + Enterprise-Support. Tool ist Marketing, nicht Produkt.

Lücke 2 — Consulting-Kern

Strukturiertes AI-Readiness Audit Framework

NICHT BAUEN (SAAS) METHODIK ENTWICKELN SOFORT UMSETZBAR

Problem: Keine Agentur bietet heute einen strukturierten, reproduzierbaren Audit-Prozess der ein Design System gegen AI-Readiness-Kriterien bewertet. Nicht als SaaS — als Methodik und Framework, das WeSlam für jeden Enterprise-Kunden durchführt.

Was fehlt: Ein standardisiertes Audit-Playbook mit 40+ Checkpoints (Token-Struktur, Code Connect Coverage, MCP-Config, Component-Naming, etc.) das in 2 Wochen durchgeführt werden kann und einen Score + priorisierten Backlog liefert.

Lücke 4 — Achtung, schwierig

Warum Hybrid der richtige Weg ist

1
Jetzt — Monate 1–3

Consulting First: 3–5 Pilot-Audits

Bevor WeSlam irgendetwas baut: Manuelle Audits bei 3–5 Unternehmen durchführen. Das Audit-Framework entsteht durch echte Arbeit — nicht durch Theorie. Outcome: CHF 60–120k Umsatz + Erkenntnis was wirklich weh tut.

💡

Warum erst Consulting, dann Tool?

Die repetitivste, schmerzhafteste Arbeit in den manuellen Audits = der Kandidat für Toolification. Ohne die Consulting-Erfahrung baut man das falsche Tool.

2
Parallel — Monate 2–4

Open Source: Design System Drift Detector

Einen internen GitHub Action bauen der im Consulting genutzt wird. Wenn er funktioniert: Open Source auf GitHub. Ziel: Community-Aufbau, Thought Leadership, Inbound-Leads. Das Tool kostet CHF 20–30k zu bauen — und zahlt sich durch 3–5 Enterprise-Inbounds zurück.

Was der Drift Detector macht:
Auf jedem PR: scannt neuen Code auf hardcoded Farben/Spacings, überprüft gegen Token-Map des Design Systems, flagged Abweichungen als GitHub-Check. Konfigurierbar pro Framework (React/Angular/Vue).
3
Monat 6+

Entscheidung: SaaS oder nicht?

Nach 6 Monaten Consulting + Open Source: Datenbasierte Entscheidung ob der Drift-Detector als SaaS-Produkt skaliert werden soll. Kriterien: Anzahl GitHub Stars, Inbound-Leads, was Kunden bereit wären zu zahlen.

🚫

Was WeSlam NICHT tun sollte

Ein vollständiges SaaS-Produkt á la Supernova bauen. Das kostet CHF 200–400k, dauert 12+ Monate und tritt gegen ein gut-finanziertes Team mit Vorsprung an. Die Energie geht besser in Consulting-Tiefe und einen fokussierten Open-Source-Beitrag.

Swiss Advantage — Kritische Analyse

Wieso nicht
offshore?

Der ehrlichste Teil dieser Analyse. Was ist wirklich verteidigbar, was ist nur Marketing? Und welche Argumente würde ein rationaler CFO akzeptieren?

🎯

Challenge: Warum nicht eine günstigere Agentur aus Polen oder Indien?

Technisch gesehen kann ein gut-organisiertes Remote-Team in Polen oder Indien denselben Code Connect Setup durchführen. Tokens Studio, Style Dictionary, Figma — alles Cloud, alles remote-tauglich. Die Arbeit ist ~90% async. Was ist also der echte Unterschied?

🏆 Stärkstes Argument: Pre-Approved Vendor Status

WeSlam ist bereits Lieferant bei Swisscom, Migros, Homegate und weiteren. Ein neuer Offshore-Anbieter muss durch den Vendor-Approval-Prozess grosser Schweizer Unternehmen — das dauert 6–18 Monate, kostet Zeit und ist risikobehaftet. WeSlam kann morgen ein Proposal schicken. Das ist kein Marketing-Argument — das ist harter Wettbewerbsvorteil.

🏆 nDSG / Datenschutz für strategisch sensible Systeme

Ein Design System enthält mehr als nur Farben: Component-Architektur, Produkt-Roadmap (welche Features sind im Aufbau), Brand-Strategie, UX-Entscheidungen. Schweizer Unternehmen, insbesondere die FINMA-regulierten (Banken, Versicherungen), haben Legal-Teams die externe Cloud-Zugänge auf Figma-Repos und Codebases kritisch prüfen. Eine Schweizer Firma unter CH-Recht ist ein 100x einfacherer Vendor-Review als ein Anbieter aus Indien oder Polen.

🏆 Physische Präsenz und Embedded Work

Die besten Design System Migrations passieren eingebettet — 1–2 Tage pro Woche vor Ort beim Kunden. Change Management, Stakeholder-Alignment, Workshops. Das ist mit einem Remote-Team aus Polen möglich — aber deutlich schwieriger und teurer wenn man die Travel-Kosten einrechnet. WeSlam kann in Zürich oder Genf am Dienstag vor Ort sein.

✅ Sprache und Kontext

Schweiz hat 4 Sprachregionen. Für viele Kunden (SBB, Swisscom, öffentliche Hand) ist Deutsch als Arbeitssprache entscheidend. Technische Dokumentation in Deutsch, Workshops auf Deutsch, Slack-Kommunikation auf Deutsch. Das klingt trivial — aber ein Offshore-Team, das alles auf Englisch liefert, schafft Reibung die akkumuliert.

✅ Bestehende Kundenbeziehungen = kein Sales-Funnel

Für einen neuen Service bei einem bestehenden Kunden braucht es kein RFP, keinen langen Sales-Prozess, keine Referenzprüfung. WeSlam geht ins nächste Quarterly Review und pitcht einen neuen Service. Das ist unbezahlbar.

❌ "Schweizer Qualität"

Generisches Marketing. Ein Offshore-Team aus Estland oder Polen kann genauso guten Code schreiben. "Swiss quality" ist kein Argument das einen rationalen Einkäufer überzeugt.

❌ "Wir verstehen die Schweizer Unternehmenskultur"

Zu schwach. Viele internationale Agenturen arbeiten seit Jahren mit Schweizer Unternehmen. Kulturverständnis ist ein Soft-Argument, das ein Preisunterschied von 40% schnell überwältigt.

❌ "Gleiche Zeitzone"

Westeuropa-Timezones sind nur 1–2 Stunden Unterschied. Minimal. Nicht überzeugend.

❌ "Die Arbeit ist zu komplex für Offshore"

Stimmt nicht mehr. Design System Arbeit (Token-Restrukturierung, Code Connect Setup) ist handwerklich anspruchsvoll aber nicht magisch. Gut organisierte Offshore-Teams können das — für 30–50% weniger.

⚠️ "Gesetzliche Preferenz für Schweizer Anbieter"

Halb-wahr. Im öffentlichen Sektor gibt es unter dem WTO GPA keine legale Preferenz für lokale Anbieter. Im privaten Sektor gibt es keine gesetzliche Regelung — aber FINMA und nDSG schaffen faktische Hürden für bestimmte Datenzugänge.

Die Moat-Strategie: Was wird schwer zu kopieren?

Moat 1

Tiefe Swiss Enterprise Referenzen

Wer SBB und Swisscom als Case Studies hat, ist unschlagbar für den nächsten Kunden in diesem Segment. Niemand fragt "warum nicht offshore" wenn der Pitch mit "wir haben das für SBB gemacht" beginnt.

Moat 2

Das Audit-Framework als IP

Ein standardisiertes, tiefes Audit-Framework (40+ Checkpoints, Scoring-Methodik, Repair-Playbooks) ist nicht öffentlich kopierbar. Es lebt in WeSlams Team-Erfahrung und wächst mit jedem Projekt.

Moat 3

Kombination Design + GenAI + Code

WeSlam ist einzigartig an der Schnittstelle UX-Design, GenAI-Integration und Frontend-Development. Kein einzelner Spezialist kann das abdecken. Kein Offshore-Team baut diese Kombination in 12 Monaten.

Moat 4

Open Source Community + Thought Leadership

Ein gut-gemachter Open Source Drift-Detector auf GitHub schafft passive Anerkennung. Talks, Blogposts, "AI-Ready Design System" als Begriff den WeSlam in der Schweiz besetzt.

Risiken — ehrlich

Accenture oder EY baut diese Practice

Grosse Consultancies haben die Beziehungen und können mehr Leute einsetzen. WeSlam muss schneller sein als die. 2–3 Referenzkunden in 2026 sind die Antwort.

Figma "löst" das Problem nativ

Figma baut die Infrastruktur — MCP, Code Connect, AI Mapping. Wenn das in 2 Jahren "plug and play" ist, fällt ein Teil des technischen Aufwands weg. Aber: Change Management, Team-Enablement, Governance bleiben immer menschliche Arbeit.

Buying Center ist unklar

Sitzt das Budget beim Head of Design oder beim CTO? Ist es ein Design-Budget oder ein AI-Transformation-Budget? Das muss WeSlam in den ersten Quickchecks herausfinden.

Finales Verdict

Die ehrliche
Empfehlung.

Zur Tool-Frage

Kein SaaS-Produkt bauen. Supernova hat $9.2M und einen Vorsprung. WeSlam sollte sich nicht in eine Tool-Konkurrenz begeben, die es nicht gewinnen kann.

Stattdessen: Consulting als Kerngeschäft. Das Audit-Framework als interne Methodik die sich mit jedem Projekt verbessert. Und parallel einen schlanken Open-Source GitHub Action (Drift-Detector) der Expertise demonstriert und Leads generiert — ohne gegen Supernova zu konkurrieren, weil er downstream und developer-focused ist.

Zum Swiss Advantage

Der echte Moat ist nicht "Schweizer Qualität" — das ist ein schlechtes Argument das jeder rationaler Käufer mit einem 40%-Preisunterschied überwältigen kann.

Der echte Moat ist: Bestehende Vendor-Beziehungen, nDSG-Compliance by default, physische Embedded-Präsenz, und die Kombination von GenAI + Design + Dev die kein Offshore-Team schnell replizieren kann.

Aber: Für neue Kunden ohne bestehende Beziehung ist dieser Vorteil schwächer. Deshalb ist der schnellste Weg zu 2–3 Marquee Case Studies (z.B. SBB, Swisscom) die wichtigste strategische Priorität.
