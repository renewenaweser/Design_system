Das Problem

Warum AI-Coding ohne Design-System-Guardrails scheitert

Kern-Problem

AI-generierter Code respektiert keine Design Systems

Wenn Cursor, Copilot oder Claude Code auf einem PR arbeitet, weiss heute niemand ob der generierte Code die Design-System-Tokens und Komponenten korrekt nutzt. Statt var(--color-primary-500) schreibt die AI #3B82F6. Statt var(--spacing-4) steht 16px. Das passiert hundertfach am Tag.

62% der Entwickler berichten Rework wegen inkonsistentem AI-Code
20–30% Produktivitätsgewinn durch AI-Coding — aber nur wenn Code konsistent bleibt
0 generische Open-Source-Tools die Design-System-Drift auf PRs prüfen
🔥

Die Drift-Kaskade

Hardcoded Werte → Inkonsistente UI → Designer meldet Bug → Dev fixt manuell → Nächster AI-generierter PR wiederholt den Fehler → Design System wird ignoriert → Team fällt zurück auf manuelle Reviews. Der gesamte AI-Productivity-Gain geht verloren.

Die Lücke

Jedes existierende Tool ist entweder (a) an ein spezifisches Design System gebunden, (b) nur CSS, nicht JS/TSX, oder (c) keine CI/CD-Integration. Es gibt kein generisches, framework-agnostisches, token-map-konfigurierbares Open-Source-Tool das auf PR-Level Design-System-Compliance prüft.

User Stories

Die drei Personas und ihre Kernbedürfnisse

Persona 1 — Frontend Developer
🧑‍💻 Frontend Developer
Als Frontend-Dev will ich auf meinem PR sofort sehen welche Stellen hardcoded Werte statt Design Tokens nutzen, damit ich sie fixen kann bevor der Review startet.
Acceptance Criteria:
• GitHub Check erscheint innerhalb von 60 Sekunden nach PR-Öffnung
• Inline-Annotations zeigen exakt die Zeile und den vorgeschlagenen Token
• False-Positive-Rate unter 10% bei korrekt konfigurierter Token-Map
🧑‍💻 Frontend Developer (AI-Coding)
Als Dev der Cursor/Copilot nutzt, will ich dass AI-generierter Code automatisch auf Design-System-Compliance geprüft wird, ohne dass ich jede Zeile manuell reviewen muss.
Acceptance Criteria:
• Erkennt typische AI-Patterns (hardcoded Tailwind-ähnliche Werte, inline styles)
• Summary-Kommentar auf dem PR zeigt Gesamtscore (X violations, Y auto-fixable)
• Optionaler Strict Mode blockt Merge bei Critical Violations
Persona 2 — Design System Lead
👩‍🎨 Design System Lead
Als DS Lead will ich wissen wie konsistent unser Design System über alle Repos hinweg genutzt wird — und ob die Adoption mit der Zeit steigt oder sinkt.
Acceptance Criteria:
• Wöchentlicher Trend-Report (Violations pro Woche, Top-10 Offenders)
• Dashboard-Daten via JSON-Export oder GitHub API
• Filtermöglichkeit nach Team / Repo / Token-Kategorie
👩‍🎨 Design System Lead
Als DS Lead will ich die Token-Map selbst konfigurieren können, ohne den Source Code des Tools anzufassen.
Acceptance Criteria:
• YAML-Config-Datei im Repo-Root (.driftrc.yml)
• Importiert W3C DTCG JSON direkt als Token-Source
• Custom-Rules für spezifische Properties definierbar
Persona 3 — Head of Engineering
👔 Head of Engineering
Als Head of Engineering will ich sicherstellen dass unsere AI-Coding-Investition (Cursor Lizenzen, Copilot Enterprise) nicht zu technischem Debt führt weil generierter Code die UI-Konsistenz untergräbt.
Acceptance Criteria:
• Quantifizierbarer ROI: "Design-System-Compliance von 64% auf 92% in 3 Monaten"
• Kein Vendor Lock-in (Open Source, self-hosted)
• Setup in unter 30 Minuten pro Repo

# .github/workflows/drift-check.yml name: Design System Drift Check on: [pull_request] jobs: drift-check: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: weslam/drift-detector@v1 with: config: ".driftrc.yml" token: ${{ secrets.GITHUB_TOKEN }}

Tech Stack

Runtime

Node.js + TypeScript

Maximale Kompatibilität mit dem Frontend-Ökosystem. TypeScript für Type Safety. Läuft nativ in GitHub Actions ohne Docker-Overhead.

Parsing

PostCSS + AST Parser

PostCSS für CSS/SCSS. @babel/parser oder ts-morph für JSX/TSX. Regex-Fallback für Tailwind arbitrary values. Modular: je ein Parser pro Sprache/Framework.

Token Matching

Fuzzy Color Matching

Exact Match (hex → token), Nearest Match (Delta-E Color Distance für nahe Farben), Spacing Scale Match (16px → spacing-4 wenn spacing-scale = 4px). Konfigurierbarer Threshold.

Output

GitHub Checks API

@actions/core für Annotations. @octokit/rest für PR Comments. GitHub Suggestions Format für Auto-Fix. Keine externen Dependencies ausser GitHub.

⚡

Performance-Ziel

Unter 60 Sekunden für einen PR mit 50 geänderten Dateien. Nur geänderte Dateien scannen (git diff), nicht das gesamte Repo. Token-Map einmal laden und im Memory halten.

Risiken & Mitigations

Was schiefgehen kann und wie wir gegensteuern

Risiko 1 — Hoch

False Positives frustrieren Entwickler

Zu viele falsche Alarme führen dazu dass Teams den Check deaktivieren. Jeder Linter stirbt an zu hoher False-Positive-Rate.

Mitigation

Konservative Default-Config (nur High-Confidence Matches). Granulare Ignore-Rules. "Confidence Score" auf jedem Finding. Schneller Feedback-Loop mit erstem Kundenprojekt.

Risiko 2 — Mittel

Supernova oder Figma baut das Feature

Supernova ($9.2M) könnte CI/CD-Checks als Feature launchen. Figma's native DTCG-Export (Nov 2026) könnte den Token-Pipeline-Bedarf reduzieren.

Mitigation

Open Source = kein Revenue-Risiko. Der Drift Detector kostet CHF 20–30k, nicht CHF 500k. Selbst wenn Supernova es baut, haben wir 6+ Monate Vorsprung und das Consulting-Positioning. Worst case: Das Tool bewies WeSlam's Expertise und generierte Leads.

Risiko 3 — Mittel

Token-Map-Konfiguration zu komplex

Enterprise Design Systems haben oft kein sauberes W3C DTCG JSON. Die Konfiguration könnte zu aufwändig sein für Self-Service-Adoption.

Mitigation

Mehrere Input-Formate (DTCG JSON + CSS Custom Properties + SCSS Vars). "Auto-Detect" Modus der Token-Files im Repo findet. Starter-Templates für gängige Setups. Und: Die Konfiguration IST das Consulting-Upsell — WeSlam hilft beim Setup.

Risiko 4 — Niedrig

Wartungsaufwand übersteigt Nutzen

Open-Source-Projekte brauchen Maintenance: Issues, PRs, neue Framework-Versionen, Breaking Changes in GitHub Actions API.

Mitigation

Bewusst schlanker Scope (kein Feature Creep). Modulare Architektur (Parser sind austauschbar). Community-PRs akzeptieren für Framework-spezifische Parser. Max. 4h/Woche Maintenance-Budget einplanen.

Bottom Line

Das grösste Risiko ist nicht technisch — es ist Adoption. Ein perfekt gebautes Tool das niemand benutzt hat keinen Wert. Deshalb: Erst intern auf Kundenprojekt validieren, dann open-sourcen. Die ersten 3 "Kunden" sind WeSlam-Bestandskunden die das Tool gratis bekommen. Ihr Feedback bestimmt ob und wie es veröffentlicht wird.
