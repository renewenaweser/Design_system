Swiss Advantage — Kritische Analyse

Wieso nicht
offshore?

Der ehrlichste Teil dieser Analyse. Was ist wirklich verteidigbar, was ist nur Marketing? Und welche Argumente würde ein rationaler CFO akzeptieren?

🎯

Challenge: Warum nicht eine günstigere Agentur aus Polen oder Indien?

Technisch gesehen kann ein gut-organisiertes Remote-Team in Polen oder Indien denselben Code Connect Setup durchführen. Tokens Studio, Style Dictionary, Figma — alles Cloud, alles remote-tauglich. Die Arbeit ist ~90% async. Was ist also der echte Unterschied?

🏆 Stärkstes Argument: Pre-Approved Vendor Status

WeSlam ist bereits Lieferant bei Swisscom, Migros, Homegate und weiteren. Ein neuer Offshore-Anbieter muss durch den Vendor-Approval-Prozess grosser Schweizer Unternehmen — das dauert 6–18 Monate, kostet Zeit und ist risikobehaftet. WeSlam kann morgen ein Proposal schicken. Das ist kein Marketing-Argument — das ist harter Wettbewerbsvorteil.

🏆 nDSG / Datenschutz für strategisch sensible Systeme

Ein Design System enthält mehr als nur Farben: Component-Architektur, Produkt-Roadmap (welche Features sind im Aufbau), Brand-Strategie, UX-Entscheidungen. Schweizer Unternehmen, insbesondere die FINMA-regulierten (Banken, Versicherungen), haben Legal-Teams die externe Cloud-Zugänge auf Figma-Repos und Codebases kritisch prüfen. Eine Schweizer Firma unter CH-Recht ist ein 100x einfacherer Vendor-Review als ein Anbieter aus Indien oder Polen.

🏆 Physische Präsenz und Embedded Work

Die besten Design System Migrations passieren eingebettet — 1–2 Tage pro Woche vor Ort beim Kunden. Change Management, Stakeholder-Alignment, Workshops. Das ist mit einem Remote-Team aus Polen möglich — aber deutlich schwieriger und teurer wenn man die Travel-Kosten einrechnet. WeSlam kann in Zürich oder Genf am Dienstag vor Ort sein.

✅ Sprache und Kontext

Schweiz hat 4 Sprachregionen. Für viele Kunden (SBB, Swisscom, öffentliche Hand) ist Deutsch als Arbeitssprache entscheidend. Technische Dokumentation in Deutsch, Workshops auf Deutsch, Slack-Kommunikation auf Deutsch. Das klingt trivial — aber ein Offshore-Team, das alles auf Englisch liefert, schafft Reibung die akkumuliert.

✅ Bestehende Kundenbeziehungen = kein Sales-Funnel

Für einen neuen Service bei einem bestehenden Kunden braucht es kein RFP, keinen langen Sales-Prozess, keine Referenzprüfung. WeSlam geht ins nächste Quarterly Review und pitcht einen neuen Service. Das ist unbezahlbar.

❌ "Schweizer Qualität"

Generisches Marketing. Ein Offshore-Team aus Estland oder Polen kann genauso guten Code schreiben. "Swiss quality" ist kein Argument das einen rationalen Einkäufer überzeugt.

❌ "Wir verstehen die Schweizer Unternehmenskultur"

Zu schwach. Viele internationale Agenturen arbeiten seit Jahren mit Schweizer Unternehmen. Kulturverständnis ist ein Soft-Argument, das ein Preisunterschied von 40% schnell überwältigt.

❌ "Gleiche Zeitzone"

Westeuropa-Timezones sind nur 1–2 Stunden Unterschied. Minimal. Nicht überzeugend.

❌ "Die Arbeit ist zu komplex für Offshore"

Stimmt nicht mehr. Design System Arbeit (Token-Restrukturierung, Code Connect Setup) ist handwerklich anspruchsvoll aber nicht magisch. Gut organisierte Offshore-Teams können das — für 30–50% weniger.

⚠️ "Gesetzliche Preferenz für Schweizer Anbieter"

Halb-wahr. Im öffentlichen Sektor gibt es unter dem WTO GPA keine legale Preferenz für lokale Anbieter. Im privaten Sektor gibt es keine gesetzliche Regelung — aber FINMA und nDSG schaffen faktische Hürden für bestimmte Datenzugänge.

1. Token-Struktur (W3C-konform)

Das Fundament — ohne das nichts anderes funktioniert

Was es ist

Design Tokens = die Atome eures Design Systems

Typischer Score: 2 / 12.5

Design Tokens sind die kleinsten Einheiten eures Design Systems: Farben, Abstände, Schriftgrössen, Border-Radien, Schatten. Sie sind die "Single Source of Truth" — der Ort wo definiert wird, dass euer Primärblau #0066CC ist und euer Standard-Spacing 16px.

W3C DTCG ist der neue, seit Oktober 2025 stabile Standard der definiert wie diese Tokens in einem JSON-Format geschrieben werden — damit jedes Tool (Figma, Style Dictionary, Supernova, euer Code) sie lesen kann. Denk an CSV für Tabellen — DTCG ist das universelle Format für Design-Entscheidungen.

❌ Vorher (typisch)

Tokens existieren als Figma-Variablen, aber nur in Figma. Kein JSON-Export. Naming ist inkonsistent: mal blue-500, mal primary, mal brand-color. Dev-Team hat eigene CSS-Variablen die nicht mit Figma übereinstimmen. Jeder definiert Farben anders.

✅ Nachher

Ein einziges Token-JSON im W3C DTCG Format. 3-stufige Hierarchie: Primitive → Semantic → Component. Figma exportiert automatisch. Dev-Code konsumiert dasselbe JSON via Style Dictionary. Eine Wahrheit, zwei Outputs.

// W3C DTCG Standard — so sieht ein Token-File aus { "color": { "primitive": { "blue-500": { "$value": "#0066CC", "$type": "color" }, "blue-600": { "$value": "#004C99", "$type": "color" } }, "semantic": { "primary": { "$value": "{color.primitive.blue-500}", "$type": "color" }, "primary-hover": { "$value": "{color.primitive.blue-600}", "$type": "color" } } }, "spacing": { "xs": { "$value": "4px", "$type": "dimension" }, "sm": { "$value": "8px", "$type": "dimension" }, "md": { "$value": "16px", "$type": "dimension" } } }
🔧 Was WeSlam konkret macht
1
Token-Inventar erstellen: Alle bestehenden Tokens aus Figma und Code extrahieren. Duplikate und Inkonsistenzen finden. Wie viele Blau-Töne hat euer System wirklich? (Antwort bei den meisten: 12–25 statt der geplanten 5.)
2
3-Stufen-Hierarchie aufbauen: Primitive Tokens (die rohen Werte), Semantic Tokens (was sie bedeuten: "primary", "error", "surface"), Component Tokens (Button-spezifisch). Die AI braucht die semantische Ebene um zu wissen was gemeint ist, nicht nur was der Wert ist.
3
W3C DTCG JSON exportieren: Figma-Variablen in das standardisierte JSON-Format überführen. Tokens Studio Plugin konfigurieren oder Figma's nativen DTCG-Export (ab Nov 2026) vorbereiten. Ergebnis: ein tokens.json das jedes Tool lesen kann.
4
Style Dictionary Pipeline einrichten: Das Token-JSON automatisch in CSS Custom Properties, SCSS Variables, iOS/Android-Formate transformieren. Ein npm run build:tokens Befehl der alles generiert. Kein manuelles Copy-Paste mehr.

2. Code Connect Setup

Die Brücke zwischen Figma-Design und echtem Code

Was es ist

Figma-Komponenten mit echtem Codebase-Code verknüpfen

Typischer Score: 0.5 / 12.5

Ohne Code Connect zeigt Figma Dev Mode generischen, auto-generierten Code. Mit Code Connect sieht der Entwickler (und die AI!) den echten Import-Pfad und die echten Props eurer Komponente. Statt <div style="background: blue"> zeigt es <Button variant="primary" size="lg" />.

Für AI-Coding ist das der wichtigste Einzelschritt. Ohne Code Connect hat die AI keinen Hinweis dass eure Codebase schon eine Button-Komponente hat — und baut eine neue von Grund auf.

❌ Ohne Code Connect

AI sieht Figma-Design → generiert: div mit inline-styles, hardcoded Farben, eigene Border-Radien. Kein Import aus der Component Library. Jede generierte Seite ist ein Unikat.

✅ Mit Code Connect

AI sieht Figma-Design → erkennt "das ist ein Button/primary/lg" → generiert: import { Button } from '@company/ui' mit korrekten Props. Konsistenter Code, keine Duplikate.

// Code Connect File — verknüpft Figma-Komponente mit Code // Datei: src/components/Button/Button.figma.tsx import figma from "@figma/code-connect" import { Button } from "./Button" figma.connect(Button, "https://figma.com/file/xxx/Button", { props: { variant: figma.enum("Variant", { Primary: "primary", Secondary: "secondary", Ghost: "ghost" }), size: figma.enum("Size", { Small: "sm", Medium: "md", Large: "lg" }), label: figma.string("Label"), disabled: figma.boolean("Disabled") }, example: (props) => `<Button variant="${props.variant}" size="${props.size}" disabled={${props.disabled}}>${props.label}</Button>` })
🔧 Was WeSlam konkret macht
1
Top-20-Komponenten identifizieren: Welche 20 Komponenten werden am häufigsten verwendet? (Button, Input, Card, Modal, Avatar, Badge, ...) Diese decken typischerweise 80% aller UI-Patterns ab.
2
Code Connect Files schreiben: Für jede der Top-20-Komponenten eine .figma.tsx Datei erstellen. Property-Mapping: Figma-Varianten → Code-Props. Einmal aufgesetzt, automatisch aktuell.
3
CLI einrichten und publishen: npx figma connect publish in die CI/CD-Pipeline einbauen. Bei jedem Merge auf Main werden die Mappings automatisch an Figma gepusht. Kein manueller Sync.
4
Team enablen: Workshop mit dem Design-System-Team: Wie schreibt man neue Code Connect Files? Wie updated man Mappings wenn sich Props ändern? Playbook mit Templates und Best Practices.

3. MCP Server Konfiguration

Der direkte Draht zwischen Figma und AI-Coding-Tools

Was es ist

Figma's MCP Server = AI-Agents können Figma "lesen"

Typischer Score: 0 / 12.5

MCP (Model Context Protocol) ist ein offener Standard der AI-Agents erlaubt, mit externen Tools zu kommunizieren. Figma hat 2025 einen eigenen MCP-Server released. Das bedeutet: Cursor, Claude oder Copilot können direkt auf euer Figma-File zugreifen — Komponenten lesen, Styles extrahieren, Design-Kontext verstehen.

Einfach erklärt: Bisher musste der Entwickler Screenshots von Figma machen und dem AI-Tool zeigen. Mit MCP kann die AI das Figma-File selbst öffnen, die Komponenten verstehen, und Code generieren der genau zum Design passt.

❌ Ohne MCP

Dev nimmt Screenshot von Figma → Paste in Cursor → AI "rät" anhand von Pixeln was gemeint ist → generiert ungefähren Code. Keine Ahnung von Tokens, Varianten, Spacing-System.

✅ Mit MCP + Code Connect

Dev sagt "implementiere das Login-Formular aus Figma Frame XY" → AI fragt MCP Server → bekommt: Komponenten-Liste, Token-Werte, Code Connect Snippets → generiert exakten Code mit richtigen Imports.

// Cursor MCP Config — .cursor/mcp.json { "mcpServers": { "figma": { "command": "npx", "args": ["-y", "figma-developer-mcp", "--figma-api-key=YOUR_KEY"], "env": { "FIGMA_API_KEY": "figd_xxx" } } } }
🔧 Was WeSlam konkret macht
1
MCP Server einrichten: Figma API-Key generieren, MCP-Config für Cursor/Claude/VS Code erstellen, auf jedem Dev-Rechner ausrollen. Setup-Script für einfaches Onboarding neuer Devs.
2
Code Connect optimieren für MCP: Der MCP-Server liefert nur dann guten Output wenn Code Connect sauber konfiguriert ist. Wir stellen sicher, dass die AI CodeConnectSnippet-Blöcke mit korrekten Imports und Props bekommt.
3
Custom Instructions hinzufügen: In der Code Connect UI können pro Komponente zusätzliche AI-Anweisungen hinterlegt werden: "Nutze immer die Accessible-Variante", "Dieser Button braucht immer ein Icon links". Die AI liest diese bei jedem MCP-Call.
4
Security klären: Figma API-Keys in Secret-Management (GitHub Secrets, 1Password). Für Schweizer Enterprises: prüfen ob Figma-API-Zugriff mit internem Security-Policy kompatibel ist (nDSG, Daten fliessen über Figma's API).

4. AI-Agent Dokumentation

Spielregeln für die AI — damit sie euer System respektiert

Was es ist

Component Rules & AI Guidelines

Typischer Score: 1 / 12.5

AI-Agents wie Cursor und Claude können "System Prompts" oder "Rules Files" lesen — Textdateien die definieren wie die AI in eurem Projekt Code schreiben soll. Ohne diese Regeln generiert die AI generischen Code. Mit ihnen generiert sie Code der eurer Design-System-Philosophie folgt.

Denk an ein Onboarding-Dokument für einen neuen Entwickler — aber für die AI. "Bei uns nutzen wir immer semantische Tokens. Wir wrappen nie einen Button in einen Link. Wir nutzen immer die Stack-Komponente für Layouts."

# .cursor/rules/design-system.mdc # Cursor Rules File — wird bei jedem Prompt mitgelesen ## Design System Rules Du arbeitest mit dem @company/ui Design System. ### Farben - Nutze IMMER semantische Tokens: `var(--color-primary)`, nie `#0066CC` - Für Hintergründe: `var(--color-bg-*)`, für Text: `var(--color-text-*)` - NIEMALS hardcoded Farben, auch nicht für Grau-Töne ### Komponenten - Button: immer aus `@company/ui` importieren, nie selbst bauen - Layouts: immer `Stack` oder `Grid` nutzen, nie raw flexbox - Icons: nur aus `@company/icons`, nie inline SVG ### Spacing - Nutze die Spacing-Scale: xs(4), sm(8), md(16), lg(24), xl(32) - gap, padding, margin: immer als Token-Variable - Kein `margin-top` auf Komponenten — nutze `Stack` mit `gap` ### Accessibility - Jeder Button braucht ein `aria-label` wenn nur Icon - Farb-Kontraste: mindestens WCAG AA (4.5:1 für Text)
🔧 Was WeSlam konkret macht
1
Design-System-Regeln erfassen: Workshop mit Design- und Dev-Team. Welche Regeln gelten heute implizit? Was steht nirgends geschrieben? Was führt immer wieder zu Review-Kommentaren? Diese "unwritten rules" formalisieren.
2
Rules-Files schreiben: Cursor .mdc-Rules, Claude CLAUDE.md, Copilot Instructions-File. Jedes AI-Tool hat sein eigenes Format — wir erstellen alle drei damit das gesamte Team abgedeckt ist.
3
Component-spezifische Guidelines: Nicht nur globale Regeln, sondern pro-Komponente: "Dialog: immer mit Portal rendern. Table: immer mit virtualisierten Rows über 50 Einträge. Form: immer React Hook Form mit Zod Schema."
4
Ins Repo committen: Rules-Files leben im Repo, nicht in persönlichen Einstellungen. Jeder Dev (und jede AI) hat automatisch Zugriff. Versioniert, reviewbar, updatebar. Teil des normalen PR-Prozesses.

5. Component-Naming (Figma ↔ Code)

Wenn Figma "CTA/Large" sagt aber der Code "ButtonPrimary" heisst

Was es ist

Identische Namen in Design und Code

Typischer Score: 5 / 12.5

Das klingt trivial — ist es aber nicht. In Figma heisst die Komponente "CTA / Primary / Large". Im Code heisst sie ButtonPrimary oder PrimaryButton oder Btn. Die AI kann diese Verbindung nicht herstellen. Sie sieht zwei verschiedene Dinge — und baut eine dritte Variante.

Naming-Konsistenz ist der Unterschied zwischen "AI versteht unser System" und "AI generiert Chaos". Jeder Namens-Mismatch zwischen Figma und Code ist ein Punkt an dem die AI scheitert.

❌ Typische Realität

Figma: "Input / Text / Default"
Code: TextInput
Storybook: FormField
CSS-Klasse: .input-text-field

4 Namen für dasselbe Ding. Kein Mensch blickt durch — die AI schon gar nicht.

✅ Nach Harmonisierung

Figma: "TextInput"
Code: TextInput
Storybook: TextInput
CSS-Klasse: .text-input

Ein Name. Überall. Die AI erkennt sofort: Figma-Komponente = Code-Komponente.

🔧 Was WeSlam konkret macht
1
Naming-Audit: Alle Figma-Komponenten exportieren, alle Code-Komponenten listen, Side-by-Side vergleichen. Jede Abweichung dokumentieren. Bei 50+ Komponenten finden wir typischerweise 30–40 Naming-Mismatches.
2
Naming-Convention definieren: Eine Regel die für Figma UND Code gilt. z.B.: PascalCase, keine Abkürzungen, Variant als Suffix (Button, nicht Btn; TextInput, nicht TxtInp). Konvention im Design System Playbook dokumentieren.
3
Renaming durchführen: Figma-Komponenten umbenennen (Bulk-Rename Plugin). Code-Komponenten refactoren (mit Aliase für Backwards-Compatibility). Migration-Guide für Teams die alte Namen nutzen.
4
Linting-Rule einbauen: ESLint-Rule die prüft ob importierte Komponenten-Namen mit der offiziellen Naming-Convention übereinstimmen. Automatisch — nicht auf Code Review angewiesen.

6. Semantische Token-Aliase

Der Unterschied zwischen "red-500" und "color-error"

Was es ist

Tokens die sagen was sie bedeuten, nicht wie sie aussehen

Typischer Score: 3 / 12.5

Primitive Tokens beschreiben den Wert: blue-500, red-400, gray-100. Semantische Tokens beschreiben die Bedeutung: color-bg-error, color-text-primary, color-border-focus.

Für AI ist das der Unterschied zwischen "rate mal" und "ich weiss es". Wenn die AI color-bg-error sieht, weiss sie: das ist ein Error-Hintergrund. Wenn sie nur red-400 sieht, hat sie keine Ahnung ob das ein Error, ein Brand-Element, oder ein Sale-Badge ist.

// ❌ Nur primitive Tokens — AI muss raten background-color: var(--red-100); /* Ist das Error? Danger? Sale? */ color: var(--red-700); /* Welches Rot für welchen Zweck? */ border: 1px solid var(--red-300); /* Zusammengehörend? Keine Ahnung */ // ✅ Semantische Tokens — AI versteht den Kontext background-color: var(--color-bg-error); /* Klar: Error-Hintergrund */ color: var(--color-text-on-error); /* Klar: Text auf Error */ border: 1px solid var(--color-border-error); /* Klar: Error-Border */
🔧 Was WeSlam konkret macht
1
Semantische Ebene entwerfen: Kategorien definieren: Background (bg), Text (text), Border (border), Icon (icon). Zustände: default, hover, active, disabled, focus, error, success. Systematik die für das ganze System gilt.
2
Mapping erstellen: Jeder semantische Token referenziert einen primitiven: color-bg-error → red-100. Dokumentiert in einer Token-Map. Wenn sich das Rot ändert, ändert sich alles — an einer Stelle.
3
Theming vorbereiten: Semantische Tokens ermöglichen Light/Dark Mode durch einfaches Umschalten der Referenzen. color-bg-surface zeigt auf white im Light Mode und gray-900 im Dark Mode. Die AI generiert theme-fähigen Code von Anfang an.
4
Migration: Bestehendes CSS durchgehen, primitive Token-Referenzen durch semantische ersetzen. Wo var(--blue-500) steht, wird var(--color-primary). Schrittweise, file-by-file, mit Linting-Support.

7. States & Props dokumentiert

Die AI muss wissen was ein Button alles "kann"

Was es ist

Jeder Zustand und jede Eigenschaft jeder Komponente ist maschinell lesbar

Typischer Score: 4 / 12.5

Ein Button hat nicht nur ein Aussehen — er hat Varianten (primary, secondary, ghost), Grössen (sm, md, lg), Zustände (default, hover, active, focus, disabled, loading) und Slots (Icon links, Icon rechts, nur Icon).

Wenn die AI diese nicht kennt, generiert sie einen Button der nur "default/medium" kann. Das Team fixt manuell — bei jeder Seite, bei jedem PR. Die AI-Produktivität geht verloren.

❌ Undokumentierte Komponente

AI generiert: <Button>Click</Button>
Kein variant-Prop. Kein Size. Kein disabled-State. Kein Loading-Spinner. Dev muss alles manuell nachrüsten. Jedes. Mal.

✅ Vollständig dokumentiert

AI generiert: <Button variant="primary" size="lg" isLoading={submitting} leftIcon={<Save />}>Speichern</Button>
Korrekte Props, richtiger State, Icon-Slot genutzt. Production-ready.

🔧 Was WeSlam konkret macht
1
Prop-Inventar: Für jede Komponente: alle Props, alle möglichen Werte, Default-Werte, Required vs. Optional. In Figma als Variants, im Code als TypeScript-Interface. Beide müssen matchen.
2
State-Matrix erstellen: Tabelle pro Komponente: Welche Kombination von States gibt es? (z.B. Button: primary + disabled + loading). In Figma als Varianten-Baum. Im Code als Storybook-Stories.
3
Figma Variants aufräumen: Viele Design Systems haben unvollständige Variants in Figma — es fehlen States oder Grössen. Wir ergänzen die fehlenden und sorgen dafür dass Figma und Code 1:1 übereinstimmen.
4
TypeScript Interfaces als Source of Truth: Die Prop-Definitionen in TypeScript werden zur Referenz. Storybook-Stories werden auto-generiert aus den Interfaces. Die AI liest die Types und kennt automatisch alle Möglichkeiten.

8. Design-Dev Handoff Automatisierung

Schluss mit manueller Synchronisation

Was es ist

Tokens, Components und Docs werden automatisch synchronisiert

Typischer Score: 2 / 12.5

Heute bei den meisten Teams: Designer ändert eine Farbe in Figma → schreibt Slack-Nachricht an Dev → Dev sucht alle Stellen im Code → ändert manuell → vergisst 3 Stellen → nächster Sprint hat Inkonsistenzen. Dieser Handoff-Prozess ist das grösste Zeitgrab in Design-System-Teams.

Die Lösung: Eine automatisierte Pipeline. Figma → Token-Export (DTCG JSON) → Style Dictionary Transform → CSS/SCSS/JS Variablen → Git Commit → PR → Review → Merge. Kein manuelles Copy-Paste. Kein Slack. Kein "hast du das schon geändert?".

❌ Manueller Handoff
